## 当前可见 Browser 验收基线

Browser 验收必须在 Codex 内置可见浏览器或 Computer Use 中执行；Playwright、API、DOM、日志和数据库只是补充证据。测试媒体只使用 `/Volumes/App_Dev/审阅平台/test-video` 中的真实视频，禁止生成彩条占位视频。当前必验路径：

1. 真实多选后每个文件形成独立稳定行，标题/集数预填、手工编辑和全批必填门禁正确。
2. 一次点击严格顺序上传；明确部分失败后继续，最后只保留失败行，再点只重试失败行。
3. 上传成功后成功行移除；待审列表刷新失败时不重传，显示固定提示。
4. 双标签页验证首条意见隐式开始，剪辑端数秒内自动看到，无需手动刷新。
5. 当前版本意见可新增、编辑、撤回、回复和批注；剪辑员只能标记“已修改”，审阅员只能重新打开“未修改”。
6. 列表与播放导航均按未修改、已修改分组，组内按时间码、意见编号稳定排序。
7. 当前版本有至少一条意见即可上传 V2/V3，历史版本与意见只读且不复制、不映射、不自动判修复。
8. 当前版本仍有“未修改”意见时，审阅员仍可在原生确认后定稿；定稿后完全冻结。

## 产品定位

帧界成片审阅台是内网部署的轻量成片审阅工具，当前形态为独立应用，未来可作为“帧界 AI 影视生产画布平台”的成片审阅模块嵌入。

核心流程：

```text
项目管理
-> 创建成片条目
-> 上传成片原片
-> 在线播放与逐帧审阅
-> 时间码意见与画面批注
-> 点击意见/时间码/时间轴点精确回放到所属版本和帧
-> 在原成片条目下追加新版本
-> 重新审阅当前版本
-> 定稿
-> 下载单个定稿原片
-> 打包下载当前项目全部定稿原片
```

## 双入口

- `/edit` 剪辑入口：项目查看、创建、编辑，成片创建和元数据编辑，V1 批量顺序上传、追加版本、人工版本对比、只读查看意见正文和回复、标记“已修改”、查看定稿信息、单片定稿原片下载。项目与成片元数据编辑必须分别从现有项目标题栏和现有成片条目行的紧凑入口按需打开，不新增独立页面区块或资料卡片。
- `/review` 审阅入口：项目归档、恢复和安全软删除，意见新增/编辑/撤回/回复/批注，重新打开为“未修改”，一票定稿、单片下载和项目打包。不提供“开始审阅”或“要求修改”操作。
- 入口导航采用非对称可见性：`/edit` 页面只显示当前“剪辑入口”，不得渲染“成片审阅”链接；`/review` 页面同时显示“剪辑入口”和当前“成片审阅”，允许审阅方返回剪辑入口。该规则只约束导航可见性，不删除或封禁 `/review` 独立路由，也不替代服务端能力与授权校验。

入口是工作流表面，不是用户身份。入口能力、写保护、未来账号权限必须分层，
最终授权是入口能力、写保护、账号/主体权限和资源状态的交集。客户端不得提交
可信 `capabilities`、`roles`、`principal_id`、`security_context` 或
`write_guard_verified`；执行上下文必须由服务端生成。

## 强制业务原则

- 不做登录、用户、成员、角色、任务中心、通知中心、交付中心、下载中心。
- 不提供已审阅内容物理删除、覆盖历史版本或撤销定稿能力；项目删除只能是审阅入口的安全软删除，列表和常规查询面隐藏项目，历史数据、媒体和审计记录保留。唯一例外：剪辑入口可在审核开始前经二次确认物理删除重复误传的成片条目，且必须同时满足 `pending_review`、仅 1 个版本、无意见、无定稿记录、无 active finalization；审核开始后前端隐藏/禁用且后端拒绝；该操作会删除对应成片条目、单一未审版本、未被其他版本/定稿引用的文件对象和 storage root 内的物理文件。上传分片已确认清理的空会话可同时删除；未确认清理的会话必须先解除 `file_id` 绑定并保留分片引用、清理状态和配额，直到维护任务确认物理清理。
- `WRITE_GUARD_MODE` 只允许 `none`、`shared_code`、`reverse_proxy`；`shared_code` 是无账号部署下可选的写操作保护码，不是用户密码、登录账号或用户管理功能。共享码只发往验证接口，成功后依赖短期 HttpOnly Cookie，前端、日志、数据库、错误和响应不得保存码值或哈希；验证字段最多 256 个字符，请求体最多 4 KiB，失败来源状态必须过期且总量有界。
- 所有版本独立审阅：新版本不继承、不复制、不映射旧意见和旧标记。
- 历史“未修改”意见不阻止当前版本定稿。
- 精确回放只在版本内成立：所有意见卡片、意见时间码、时间轴意见点、上一条/下一条意见入口都必须通过 `ReviewPlaybackTarget` 定位，禁止只传 `timeMs` 或根据当前版本、文件名、显示版本号、数组下标推断目标。
- 精确回放必须驱动真实 HTMLVideoElement，等待 `loadedmetadata`、`canplay`、`seeked` 和可用时的 `requestVideoFrameCallback`；快速连续点击时旧媒体事件、旧请求和旧回调不得覆盖最后一次目标。
- 精确回放后只显示目标 Issue 当前 Revision 对应的 AnnotationSet；不得混合显示其他意见、其他版本或旧 Revision 的标记；默认不常驻显示当前版本全部标记。
- 定稿精确冻结 `project_ref_id`、`review_item_id`、`version_id`、`original_file_id`、`original_filename`、`file_size`、`sha256`、`media_snapshot` 和 `finalized_at`。
- 未定稿版本不得下载原片；项目包只包含当前项目已定稿原片。
- 防串项目、成片、版本、意见、标记、回复、定稿文件和项目包。

## 项目、版本和审阅规则

- 项目创建和基础信息编辑只在剪辑入口显示；项目编号、名称和说明是本地项目元数据，项目说明必须独立持久化，不得借用具有唯一身份语义的 `external_project_id`。`external_project_id` 只用于真实 host 项目关联。基础信息编辑入口附着在项目标题栏，成片标题/集数编辑入口附着在对应成片条目行，表单可使用现有 dialog 或 inline 模式按需打开，但页面不得常驻独立“项目资料”或“成片元数据”section/card；审核工作台不得展示任何元数据编辑入口或表单。项目归档、恢复和软删除只在成片审阅入口显示。归档项目仍可查看、播放、下载既有定稿和创建定稿包，但禁止继续写入审阅业务，也不得显示或执行项目删除；必须先恢复为 active 才能软删除。软删除项目从列表、详情、工作台、播放、下载和打包查询面隐藏，禁止任何后续写命令。
- 成片条目创建只在剪辑入口显示。V1 文件可多选，每个 `File` 对应稳定独立行和自己的标题/集数/错误；标题为去掉最后扩展名的文件名，集数先取“第N集”，否则仅在存在唯一 1～3 位数字候选时预填并保留前导零。任一行必填缺失时整批不开始。上传严格串行、不自动重试；明确失败保留输入并继续，成功行移除，再点只上传失败行。每个成功行同时创建 V1 并设为当前版本；`item_code` 创建后不可修改。
- 剪辑入口允许删除审核开始前的重复成片条目用于去重；一旦条目开始审核、存在意见、存在多版本或存在任何定稿记录，删除入口必须消失，后端命令必须拒绝。删除必须真实删除该条目的审阅业务行、单一未审核版本行、未被其他版本或定稿引用的 file object，并在数据库事务提交成功后删除 storage root 内对应 blob；只有清理已确认且无分片引用的上传会话才能同时删除，其他会话解除 file 绑定后继续占用配额并由维护任务回收；outbox 事件保留聚合 id 但不得继续持有指向已删除行的外键。
- 上传新版本只在剪辑入口、具体成片条目内显示；当前版本至少存在一条未删除意见即可追加，意见状态和 `changes_requested` 不是门禁，不需要 `supersede_reason`。新版本必须在同一成片条目下追加为 `max(version_no)+1`，旧版本及所属意见只读保留，不复制、不映射、不自动判定修复，新版本状态回到 `pending_review`。
- 历史版本可播放、可查看独立意见和标记、可参与人工版本对比，但不提供历史原片下载，也不产生自动问题对应、自动时间码映射或自动修复判断。
- 创建意见只允许审阅入口；第一条意见必须在同一事务内把 `pending_review` 变为 `in_review`，无显式开始审阅命令，意见编号由服务端在单个成片条目内分配。
- 当前版本在上传下一版或定稿前可持续新增、编辑、撤回、回复和批注；编辑意见或标记必须创建新 `ReviewIssueRevision`，旧 Revision 只读保留，“已修改”意见也可继续编辑。剪辑入口只能把“未修改”标记为“已修改”，审阅入口只能把“已修改”重新打开为“未修改”，服务端必须拒绝越权路径。
- 当前版本查询在页面挂载时每 2.5 秒有界轮询，卸载后停止；不使用 SSE/WebSocket。意见统一按未修改在前、已修改在后，组内按时间码、意见编号排序，列表与播放导航必须一致。
- 定稿只允许当前版本，审阅员拥有一票定稿权，“未修改”意见不阻断。仍必须满足原片存在且 SHA-256 校验通过、媒体快照完整、playback ready、不存在 active finalization，并经明确原生确认；后端仅接受 `confirmed: true`。定稿不可撤销，定稿后条目全部写命令拒绝。
- 旧 `changes_requested` 数据保留读取兼容，不需要迁移重写；其当前版本不得被锁死，应与普通未定稿当前版本一样允许意见写入、状态变更、追加版本和定稿。
- 单文件上传一旦绑定成功即视为成功；之后的待审列表刷新失败不得降级成上传失败或触发重传，必须提示“文件已上传成功，待审列表暂时刷新失败，请刷新页面查看。”
- 上传完成必须由服务端使用受限媒体探测器验证真实视频轨并生成时长、宽高和帧率；不得信任客户端声明的媒体元数据，探测器缺失、超时或媒体畸形时必须 fail closed，且错误不得回显本地路径或探测器原始输出。终态校验失败必须将上传会话标记为 aborted；分片只能在物理删除成功或确认不存在后清除数据库引用并释放配额，删除失败必须保留引用和预留供维护重试。complete 必须先用短事务提交可恢复的 finalizing lease，关闭数据库连接后完成多 GiB 拼接、SHA-256、ffprobe 和持久化，再以新短事务校验 lease 并发布；lease 过期只开放同一幂等操作 takeover，在新 lease 真正取代前原 worker 仍可发布，一旦新 lease 获胜旧 worker 必须被拒绝。活动上传的 `finalization_file_id`/已发布 `file_id` 都是孤儿扫描必须保护的数据库引用。completed/aborted 清理失败必须按有界退避重试，超过上传 TTL 且 lease 已过期的 finalizing 会话必须转入受控 abort/物理回收；过期的 initiated/receiving 会话也必须由维护任务回收。
- MP4/M4V/MOV/QT 均属于受支持的 ISO-BMFF 视频入口。浏览器 `File.type` 只是提示信息：前端可按受支持扩展名归一化为空或非规范的浏览器 MIME，但服务端仍必须同时验证允许的扩展名/MIME 组合、结构化 `ftyp` 和受限 ffprobe 的单一真实视频轨。服务端不得把少量硬编码 brand 白名单当作合法 MP4 的唯一真值；未知但结构合法的 ISO-BMFF brand 必须进入 ffprobe 终审，伪造扩展名、伪造 `ftyp`、无视频轨、畸形元数据或探测失败仍必须拒绝。
- 上传 init 必须在 PostgreSQL transaction advisory lock 下原子核算数据库持久化的全局/主体会话数与预留字节；每个会话必须按分片与完整 staging 文件同时存在的峰值预留至少 `2 * declared_size`，initiated、receiving、finalizing 以及分片物理清理未确认的 completed/aborted 都占用配额。init 的文件系统低水位门禁也必须按该峰值判断。PUT 必须先用独立短事务取得会话行锁，验证 identity/owner/status、已接收分片与本次允许的剩余声明字节，更新活动时间并提交关闭连接，再读取 body；已知 `Content-Length` 超限必须在落盘前拒绝，流式 writer 的硬上限不得大于剩余额度，body durable 后仍须在行锁下二次校验。body read、write、flush、fsync 共用总超时且阻塞 I/O 必须离开事件循环；上传会话 TTL 必须大于单次 body 总超时及清理安全余量，禁止维护任务回收仍在合法读取的请求。
- LAN 无账号部署把所有编辑客户端视为同一共享主体时，运行时必须至少允许 10 个合法上传会话和 10 个不同 upload id 的分片 PUT 同时进行；交付 admission 采用 `16/1/64`，即共享主体可占用 16 个 PUT，但同一 upload id 同时只能有一个候选，第二个重试必须在读取 body 和创建候选前拒绝。会话数、进程 admission、预留字节和磁盘低水位必须一起配置和验证。该并发保证不取消 `2 * declared_size` 预留或低水位保护，超出实际磁盘容量的组合仍须安全拒绝。initiated/receiving 的 inactivity TTL 必须明显短于 24 小时、且严格大于单次 body 总超时与清理安全余量，使刷新或断线遗留的未绑定分片在安全窗口后自动释放配额。
- Complete 的 finalizing lease 必须持久化幂等 key 哈希和规范请求哈希；lease 过期只允许同一 key、同一请求哈希的操作 takeover，不同 key 或请求必须返回幂等冲突。迁移必须把无法证明旧幂等身份的遗留 finalizing 行退回可恢复状态，不能让新 key 接管未知旧操作。
- 同一挂载页面中的 V1 创建以及 V2/V3 追加必须以 `File` 与业务字段组合维持稳定操作身份：完成上传结果和对应 command id 在不确定响应后复用，重试只补做缺失绑定或读取，不能重复创建成片或版本。页面重新载入会丢失 `File` 身份，因此必须先要求用户确认列表现状，不能把硬刷新当作恢复机制。真实栈 Playwright profile 必须显式接收本地前端 base URL 与非敏感一次性视频路径，走真实代理、后端和 PostgreSQL；它是自动化集成证据，不能替代 22 项可见 Chrome/Computer Use QA。
- 分片上传必须在 Safari 与 Chrome 中提供从已发送字节计算的单调进度；不能使用无法暴露上传字节进度的 Fetch 结果回调冒充实时进度。分片大小必须在最多 256 片和服务端单片上限内动态选择，慢速 LAN 的首片传输期间也必须持续更新既有单一底边进度线。二进制 PUT 的响应不确定时必须保留同一 upload id、part number 和操作身份并安全重试；同一页面在已失败且不再进行中的操作上重新选择相同 `name/size/type/lastModified` 的文件，只有在用有界内存逐字节确认服务端已接收前缀与新 `File` 完全一致后才可复用既有会话，前缀不一致必须 fail closed，不能把两份文件的分片拼接成混合成片，也不能无界创建新会话；两个不同 `File` 对象在上传中或完成待绑定阶段发生相同四元组碰撞时同样必须 fail closed，禁止共享 Promise 或 `file_id`。只有明确放弃的会话才可调用 abort；网络错误、超时或响应解析失败不得被伪装成确定 abort。
- 交付运行时必须常驻维护任务，独立隔离上传分片、过期下载包、数据库不可见的最终媒体/ZIP 和物理删除 tombstone 的单文件失败，失败项保留到后续周期重试且不得饿死同批其他清理；数据库批次和目录扫描都必须有固定上限，已完成物理回收的过期包必须记录回收状态，避免永久重复扫描。过期上传必须按固定上限、稳定顺序通过行锁原子 claim 并提交为 aborted，再执行物理分片删除，清空后的过期终态会话必须移除以保证后续批次可前进，上传写入/完成/终止必须使用同一会话锁边界。每次 PUT 必须使用服务端随机、独占创建的不可碰撞候选文件，禁止用客户端 Request-ID 充当文件唯一性；明确 commit 回滚必须补偿删除候选，commit 结果不确定时必须从独立数据库会话核对，已提交引用的文件不得删除，无法判定的文件保留到 TTL 回收。重传成功只能在 commit 后清理旧分片。损坏、超大、非法 UTF-8、非对象或 symlink tombstone 必须拒绝且不得越过 storage root；物理删除 tombstone 必须原子写入并持久化文件与父目录，blob 删除持久化后才能删除 tombstone。维护任务和 Alembic 启动迁移必须分别有数据库连接/statement timeout；维护任务还必须有单周期 timeout、连续错误退出、可验证 heartbeat、`degraded/error` 非健康状态和 restart policy，日志只记录状态、计数与异常类型。
- backend、一次性 migrate、维护任务和独立 package worker 必须通过交付 Compose overlay 使用同一不可变 `repository@sha256:<digest>` 应用镜像引用；四者中任何一个都不得在交付启动时保留可变 tag 或 build 配置，三个常驻服务以固定非 root UID/GID 运行。当前上传 admission limiter 是进程内状态，因此本交付必须强制单 backend replica、单 Uvicorn worker；未来水平扩容前必须引入外部协调器，禁止静默倍增主体/会话/全局并发。overlay 必须从独立 repository 和 64 位小写 digest 字段构造引用，受支持的交付 wrapper 必须在调用 Compose 前拒绝可变 tag 和非法 digest，容器引擎与启动入口再做不可变引用和运行时校验。原生 `docker compose config` 只作为结构解析门禁，不得被描述为 digest 真实性或不可变性校验。基础 Compose 的可变 tag 只允许本地构建，不能作为交付启动引用。旧数据卷 ownership 迁移不得信任应用可写 marker，不得跟随 symlink，运行期每次文件操作也必须重新 no-follow pin managed root；三个应用容器必须通过 root-only 锁串行化 ownership 检查，迁移状态必须保存在应用用户不可写的独立运行状态卷。受控恢复或 out-of-band 导入必须提供并执行一次显式强制重验，健康后关闭该开关。同一数据库不得被不同存储根的 backend 同时写入；backend 启动前必须完整审计数据库文件关联，显式 `/readyz` 也必须拒绝路径不属于当前存储合同、缺失或不是普通文件的运行时；高频容器健康探针只能执行有界的数据库连接和 Alembic revision 检查。PostgreSQL 16 必须使用命名卷、相互独立的 admin、非超级 owner/migrator 与最小 DML runtime 角色、已迁移并授权的独立测试库和仅本地必要端口；三个常驻应用服务不得接收 admin/owner 凭据，runtime 不得拥有数据库、schema 或业务对象，也不得具备 `CREATE`、`TEMP` 或 DDL 能力。runtime 对 `alembic_version` 只能 `SELECT`，对序列只能使用 `USAGE/SELECT`，不得通过迁移表 DML 或 `setval()` 改写 schema/主键状态。
- backend 必须在整个 HTTP lifespan 内用独立物理 PostgreSQL 连接持有数据库级 session advisory writer lock，第二个 runtime 在提供服务前失败，readiness 核对锁连接身份，退出时同连接核对解锁；仅显式 SQLite 测试豁免。配置的 storage/package root 必须从根开始逐级用目录 FD 与 `O_NOFOLLOW` 打开并做第二遍 device/inode 稳定性核对，中间 symlink 或替换必须 fail closed。备份恢复临时库清理前必须同时核对随机数据库身份哨兵与 owner，不能仅凭名称执行 DROP。
- 单个定稿原片和项目 ZIP 不得由前端 `fetch().blob()` 整体载入内存；HTTP 运行时必须交给浏览器原生下载流，单片原片支持标准 Range。单上传最多 256 个分片且部署配置不得放宽该上限；单包最多 100 个文件、5 GiB 原片，待构建队列、按 ZIP 预留并以成品实际字节复核的全局临时包卷配额和单项目 preparing 数都必须硬限制，视频归档不得进行无意义的高 CPU 压缩。配额必须计入所有尚未确认完成物理回收的包，包括 `failed` 终态；只有安全删除或确认不存在候选 ZIP 后才能释放预留，损坏 ready ZIP 在 replacement admission 时仍须完整计入，不能预扣尚未执行的 post-commit 删除，删除失败必须保守保留配额等待重试。项目包使用“准备中 -> 就绪 -> 下载中/失败”的两步交互：POST 必须先提交可见的 `preparing` 快照并立即返回 202，只由独立单并发 package worker 通过数据库全局锁构建；worker 必须用短事务提交随机 build lease 后关闭 Session，在事务外完成 ZIP、摘要与 fsync，再用新短事务校验同一 lease identity 并发布，takeover 成功后旧 worker 不得发布或删除新 lease 产物。worker 崩溃或超时后必须使用有上限的延迟重试，避免毒任务阻塞后续队列，超过重试上限后必须转为失败，轮询总次数和单次请求都必须有硬超时。短期签名凭据只能经 header 换取数据库原子消费、path-scoped、HttpOnly、SameSite=Strict 的一次性浏览器下载 session；同一包同一时刻只允许一个摘要校验/流式下载 lease，并有完成后冷却时间。流式响应必须按同一 lease identity 周期续期，直到响应关闭或异常；续期失败必须停止流，关闭/异常必须停止 heartbeat 并释放自身 lease，旧 lease 不得续期新 lease。凭据不得出现在 URL、持久幂等记录或过期后的前端内存。可信代理缺失合法转发协议时会话必须 fail closed；建包失败不得保留 token 或显示为下载就绪。完成的 ZIP 必须持久化自身 SHA-256；完整原片下载与 ZIP 下载在同一个固定文件描述符上完成摘要校验和流式返回，Range 请求依赖上传发布时摘要校验、不可变文件身份和启动存储审计，禁止把小 Range 放大成整文件预哈希；替换或不匹配时拒绝完整下载。清理过期包前必须核对 `package_id` 与规范 ZIP 路径绑定，活动下载 lease 未到期时不得删除。

## 2026-07-09 UI Follow-up Scope

- 当前版本意见卡片允许在审阅入口软删除意见，必须二次确认；历史版本意见始终只读且不能删除。
- 删除意见不能是假删除：合同、HTTP facade、后端命令、数据库持久化、mock adapter 和测试必须闭环；物理 issue、revision、annotation、message 记录继续保留。
- 删除后的意见从当前版本意见列表和当前版本意见计数中排除；重复删除或历史版本删除必须被拒绝。
- 工作台底部展示剧集列表，不展示版本列表；同一剧集不能因为 V1/V2 出现重复剧集项，项目详情也必须以成片条目去重。

- 版本对比区滚动进入时不能漏出或裁掉主窗口内容；左右播放器容器等高，视频按自身比例 `contain` 适配，不能拉伸变形。
